# 沙箱审查环境设计 + 吊销/事件响应流程

## 沙箱审查环境（Quarantine Review）

**原则：把每一次审查都当成在跑不可信代码。** 环境用完即焚，不留状态。

### 隔离层级（按 skill 风险分级选用，不是所有 skill 都需要最高强度）

| 强度 | 技术 | 适用场景 |
|---|---|---|
| 基础 | 一次性容器（非 root 用户、只读挂载、tmpfs 工作目录） | `risk_tier: low`，无网络声明的 skill |
| 加强 | gVisor（`runsc`）—— 在用户态拦截系统调用，不直接把 syscall 透传给宿主内核 | `risk_tier: medium`，申报了 shell 执行 |
| 最高 | Firecracker microVM —— 独立虚拟机边界，适合需要强隔离且能接受稍高的启动开销的场景 | `risk_tier: high`，申报了密钥访问或首次发布者的高风险提交 |

不管哪一级，都要叠加：seccomp 系统调用过滤、网络命名空间 + 默认拒绝出站（allowlist 例外）、非 root 用户、只读文件系统挂载（除了显式声明要写的目录）。

### 验证环境的关键设计：假凭证 + 全量记录

- 环境里放**假仓库 + canary secrets**（看起来像真密钥、真项目结构，实际是诱饵，一旦被读取/外发就能立刻触发告警）。
- 不使用任何真实的员工 token/凭证——这是最容易犯的错误：为了让 skill"能正常跑起来验证功能"，图省事塞了真实凭证进沙箱，直接让审查环境本身变成攻击面。
- 记录粒度：每一次工具调用、每一次文件读写、每一次网络请求尝试（包括被拦截的）。审查通过与否，看的不是"程序退出码"，而是这份行为日志里有没有"读取凭证类文件 + 尝试连接外部地址"这类组合信号。

## 吊销 / 弃用流程

一个 skill 从"已发布"变成"需要下架"，触发条件包括：安全审查后发现问题、事后被发现是恶意提交、依赖的上游内容被污染、维护者账号疑似被接管。

### 具体步骤

1. **标记吊销状态**：在 registry 记录里把该版本标为 `revoked`，附上原因（内部审计用，不一定要对所有用户可见细节）。
2. **索引层面下架**：`marketplace.json` 重新生成，不再包含该版本（新安装拿不到）。
3. **已安装用户的处理**（这是最容易被漏掉的一环——纯静态索引删除只挡住新安装，挡不住已经装了的）：
   - 短期：广播通知（内网公告/IM机器人）给可从审计日志（[02-lifecycle-controls.md](./02-lifecycle-controls.md) 第 8 步）查到的"已安装该版本"的用户名单，要求手动卸载/升级。
   - 中期（Phase 2/3 才具备条件）：客户端策略配置支持"blocklist"——即使本地已经装了，被 hub 标记吊销的 skill 版本在下次调用时也会被 Claude Code 拒绝加载（依赖 [03-manifest-and-runtime-enforcement.md](./03-manifest-and-runtime-enforcement.md) 里"策略下发路线"是否已经落地）。
4. **事后复盘**：把这次事件学到的东西写回 [`01-threat-model.md`](./01-threat-model.md)（新的攻击模式）或本文档（流程漏洞），复用 Trellis 自带的 `trellis-update-spec` 约定——安全 spec 也是"活文档"，每次事件都应该让下一次审查更准。

## 与架构方案的衔接

Phase 1（[`../architecture/03-deployment-and-auth.md`](../architecture/03-deployment-and-auth.md)）里提到的"纯静态托管"能覆盖上面第 2 步，但覆盖不了第 3 步的"已安装用户"处理——这是评估"什么时候必须升级到 Phase 2/3"时的一个重要判断依据：如果内网 skill 的使用规模/风险等级到了"一次误发布可能造成大范围影响"的程度，就不能只靠"静态索引删除+人工广播"这种弱吊销机制。
