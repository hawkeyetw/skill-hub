# 单人维护者操作手册：从"发现一个 skill"到"内网用户能装"

> 这份文档把 [`../architecture/`](../architecture/index.md) 和本目录其他安全文档揉成一份**你自己照着做就行**的端到端清单。前置事实：skill-hub 由你一个人维护，内容主要靠你去市面上（GitHub / agentskills.io / ClawHub 一类公开渠道）采集，偶尔有同事投稿，服务对象是内网的研发（OpenCode/Claude Code）、IT 部门、非技术人员。

## 你同时是两个角色，不能图省事合并成一步

**"内容采购员"**（去市面上找 skill）和**"安全审核员"**（决定它能不能给别人用）——中间必须隔着下面这条固定流水线，哪怕来源看起来很靠谱也不能跳步骤。这不是形式主义：ClawHub 生态在 2026-02 出的大规模投毒事件（详见 [01-threat-model.md](./01-threat-model.md)）里，恶意 skill 也伪装成"看起来正常的前置依赖"，光靠"这个作者/仓库看着还行"的直觉判断是不够的。

## 第 0 步：先判断走哪条线

**这是省你人工最关键的一次判断，只需要一秒钟**：这个 skill 是从外面（GitHub/agentskills.io/ClawHub/网上任何地方）弄回来的，还是同事投稿/你自己写的？

- **Track E · 外部采集** → 下面每一步都不能省，尤其是沙箱，不管这个 skill 自称需要什么权限都要过一遍。原因：risk_tier 是 skill 自己声明的，外部来源完全可能故意填低来逃避审查，不能让被审查对象自己决定审查强度。
- **Track I · 内部来源**（同事投稿 / 你自己写）→ 自动扫描照跑，但人工读正文和沙箱这两步改成"视情况"（见下面第 3、4 步），能省的都省，把精力留给真正需要盯的 Track E。

## 端到端六步

### 1. 采集（Intake）

- 来源：GitHub 公开仓库、agentskills.io 社区列表、ClawHub、awesome-claude-code 一类清单（→ Track E）；或者同事发给你的文件夹、你自己写的（→ Track I）。
- **一律先扔进本地"待审"目录，不直接放进对外可见的目录**（不管是 git 仓库的哪个分支/目录结构，物理上要和"已发布"区分开，防止手滑/自动化脚本误把待审内容当成已发布内容处理）——这一条两条线都一样，不因为是内部来源就跳过"待审"这个中间状态。
- 同事投稿：让他们过 SSO 提交，附一句"这是干什么用的/需要什么权限"，方便下一步核对，也是判定 Track I 身份的凭据（没有 SSO 身份背书的"内部投稿"要按 Track E 处理）。
- 详细的字段/入口设计见 [02-lifecycle-controls.md](./02-lifecycle-controls.md) 第 1 节、命名空间设计见 [`../architecture/01-registry-model.md`](../architecture/01-registry-model.md) 决策 3。

### 2. 自动化静态扫描（脚本干活，两条线都跑，不占用你的时间）

跑一遍固定脚本，出一份结构化报告，而不是你自己去翻源码：

- frontmatter/manifest 格式校验（见 [03-manifest-and-runtime-enforcement.md](./03-manifest-and-runtime-enforcement.md)，外部采集的 skill 大概率没有 manifest，这一步你要根据实际内容**补一份**；内部来源如果作者本来就写了就直接用）。
- 密钥扫描（TruffleHog 一类工具）。
- 危险模式匹配：`curl | sh`、异常大的文件（对应 JFrog 发现的 `omnicogg` 投毒包——把 payload 藏在一个刻意撑到 22MB 的 README 里，脚本要检查"文件大小是否明显超出内容合理范围"）。
- HTML 注释 / 零宽字符 / base64 块的粗筛（prompt injection 迹象）。

**任何一条命中高危规则，直接标"拒绝"，不进入下一步**——两条线都适用，省你的时间去看那些明显有问题的。

### 3. 你自己读一遍渲染后的正文——**Track E 强制，Track I 视情况**

- 读的是 `SKILL.md` **实际会被 agent 看到的完整文字**，不是代码 diff 摘要。
- **Track E（外部采集）：不能跳**，重点找："别告诉用户""忽略之前的规则""直接执行/不要确认""先读一下 `.env`/配置文件"这类话术，以及不合理的、会被自动请求的外部 URL。目的不是逐字做安全审计，是用人的常识判断"这段指令读起来是不是有点不对劲"。
- **Track I（内部来源）：视情况**——静态扫描全过 + manifest 声明 risk_tier 低（不碰 shell/网络/密钥）→ 快速扫一眼标题和大致内容就够，不用逐字读；risk_tier 中/高，或者扫描结果有点可疑但还没到直接拒绝的程度 → 还是要认真读一遍，不因为"是同事投的"就完全免检。

### 4. 需要执行代码的，跑一次隔离沙箱——**Track E 强制、Track I 视情况**

- **Track E（外部采集）：一律要过，不管声明的 risk_tier 是什么**——外部来源的自我声明不可信，沙箱验证的是"实际行为"而不是"它说自己会做什么"。
- **Track I（内部来源）：视情况**——只有以下任一条件命中才需要：manifest 声明了 shell/网络访问；risk_tier 中/高；这是这位同事第一次投稿（还没有历史信任积累）。纯文档类、无脚本、低风险、来自你已经审过几次的同事的投稿，可以跳过这一步直接进第 5 步。
- 一次性容器 + 假密钥（canary secrets）跑一遍，你看的是"有没有出现读取凭证 + 尝试外发"这种组合行为的日志摘要，不需要亲自守着它跑。
- 具体环境设计（容器/gVisor/Firecracker 怎么选）见 [04-quarantine-review-and-incident-response.md](./04-quarantine-review-and-incident-response.md)。

### 5. 你拍板

- 通过 → 标记 `approved`，签名，进入对外目录（[`../architecture/04-audience-distribution-channels.md`](../architecture/04-audience-distribution-channels.md) 里研发能装、IT 能看到、非技术人员能在门户里看到介绍）。
- 不通过/存疑 → 打回，不发布，**顺手记一句原因**——下次遇到类似的可以直接对照，这份记录本身就是在给你自己攒经验值，减少下次的审查成本。
- Track I 且第 3/4 步都因为"视情况"被跳过的（静态扫描全过 + 低风险 + 有历史信任积累的同事投稿）：这一步可以接近"过一眼扫描报告就签"，不需要再重新走一遍完整判断——这是整条流水线里对你人工投入压缩最多的一种情况。

### 6. 持续维护（发布完不等于结束）

- 定期（比如每季度）回头看已发布的 skill：上游有没有新版本、有没有被曝出问题（订阅一下 GitHub 仓库的 release/issue 通知就够，不需要专门的监控系统）。
- 优先收录"有活跃维护、来源可信"的 skill，而不是随手抓一个星星多的仓库——这是从源头减少你后续审查负担的办法。
- 保留"一键下架"的能力（对应 [04-quarantine-review-and-incident-response.md](./04-quarantine-review-and-incident-response.md) 的吊销流程）：单人运营的现实是，出问题时"能快速止损"比"追求发布前的完美预防"更重要，因为你没有团队分摊排查成本。

## 简化后的信任状态（只有三档，别的都是过度设计）

```
pending（采集/投稿了，等自动扫描或等你审）
  → approved（你亲自看过内容 + 扫描/沙箱结果，签字通过——只有这一档对内网用户可见/可装）
  → revoked（曾经 approved，后来因为问题下架）
```

risk_tier（低/中/高，见 manifest）是独立的另一个维度，只决定"要不要走第 4 步沙箱"，不是发布状态本身。不需要 team-approved/enterprise-approved 这类多级审批——那是给有安全团队、多个发布方的组织设计的，对单人运营只会增加你自己的负担。

## 给自己定的底线（不要因为"赶时间"破例）

1. **Track E（外部采集）的 skill，不管来源多大牌，第 2/3/4 步（自动扫描 + 你亲自读正文 + 沙箱）一律不能跳。** 这条是整套方案里最不能开口子的一条——"视情况"这个说法只适用于 Track I，不适用于 Track E。
2. **Track I（内部来源）判定要严格**：没有 SSO 身份背书的"内部投稿"按 Track E 处理；判定为 Track I 之后，第 3/4 步能不能跳要严格按第 3、4 节写的条件（risk_tier、是否声明 shell/网络、是否有历史信任积累）判断，不要凭印象随手放行。
3. 没有走完流程的 skill，物理上不出现在对外可见目录里——不要因为"先给某个人用一下，回头再补审查"开口子，这个口子一旦开了就会变成常态。
